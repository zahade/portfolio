Cloud-Based WebRTC Camera Streaming System

Project Objective

The primary objective of this project was to develop a self-hosted, cloud-based surveillance infrastructure that enables remote camera access across multiple networks without requiring port forwarding at client sites. Traditional IP camera systems require complex networking configurations, VPN setups, or port forwarding, creating security vulnerabilities and deployment challenges. This system eliminates these barriers by leveraging WebRTC technology and cloud-based signaling servers.

The goal was to architect a scalable, manufacturer-agnostic solution that automatically discovers cameras via RTSP streams, establishes peer-to-peer connections through TURN/STUN servers, and delivers real-time video streams to a centralized web dashboard accessible from anywhere. This infrastructure supports 20+ sites, providing secure, low-latency video surveillance without networking overhead.

Project Description

This cloud-based surveillance system consists of four core components working in concert: a camera agent application running at client sites, multiple cloud VMs hosting TURN/STUN servers for NAT traversal, a WebSocket signaling server for WebRTC negotiation, and a responsive web dashboard. The architecture enables seamless video streaming across network boundaries without exposing client infrastructure to the internet.

The camera agent, built with Python and Tkinter, runs locally at each client site and provides an intuitive GUI for adding cameras via RTSP URLs. When a camera is added, the agent automatically generates a go2rtc YAML configuration file that defines the camera source and streaming parameters. This configuration is transmitted to the cloud signaling server via WebSocket connection, which coordinates WebRTC session establishment between the camera source and remote viewers.

The system leverages TURN (Traversal Using Relays around NAT) servers hosted on cloud VMs to relay media streams when direct peer-to-peer connections aren't possible due to restrictive firewalls or symmetric NAT configurations. TURN-over-TLS implementation ensures encrypted media transmission and enables unlimited horizontal scaling by adding additional relay servers as client sites grow. The STUN (Session Traversal Utilities for NAT) servers handle initial NAT type detection and public IP discovery.

The web dashboard, built with HTML5, JavaScript, and WebRTC APIs, provides a multi-camera grid view with 4-slot configuration supporting live streaming from any registered camera. The dashboard maintains persistent WebSocket connections to the signaling server, receives camera availability updates in real-time, and establishes WebRTC peer connections on-demand when users select cameras to view. This architecture minimizes bandwidth usage by streaming only actively viewed cameras while maintaining instant availability for all configured devices.

Results & Visualizations

1. Camera Agent - Adding New Camera

Description: The camera agent application's "Add New Camera" interface running at a client site. Users input the camera name (e.g., "Camera_1") and RTSP URL with credentials (rtsp://user:password@192.168.x.x:554/cam/realmonitor?channel=1&subtype=0). When "Add Camera to Cloud" is clicked, the agent validates the RTSP connection, generates a go2rtc YAML configuration entry defining the camera source, and transmits this configuration to the cloud signaling server via secure WebSocket connection. The signaling server receives the configuration, updates its camera registry, and broadcasts availability to all connected dashboard clients. The service status indicator shows whether the camera agent's background service is running—when stopped, cameras won't sync to the cloud. This interface eliminates manual YAML editing and provides a user-friendly way for non-technical staff to onboard cameras without networking knowledge.

2. Camera Agent - Configured Cameras & Cloud Connection

Description: The camera agent's "Configured Cameras" section displays all successfully added cameras with their RTSP URLs and removal options. Below, the "Cloud Connection" panel shows critical networking details: the cloud dashboard URL, TURN server address for media relay, local API port for go2rtc control, and WebRTC port for peer connections. Once the service is started via the "Start Service" button, the agent establishes a persistent WebSocket connection to the signaling server, continuously monitoring camera availability and relaying status updates. The TURN server handles media relay when direct peer-to-peer connections fail due to symmetric NAT or firewall restrictions. The local go2rtc instance manages RTSP-to-WebRTC transcoding, converting camera streams into WebRTC-compatible formats. This architecture allows cameras on private networks (192.168.x.x) to be securely accessible from the public cloud dashboard without exposing the local network or requiring port forwarding on client routers.

3. Camera Agent - Real-Time Service Logs

Description: Real-time service logs displaying the camera agent's operational status with color-coded messages. The logs show the complete camera registration workflow: "Camera 'Camera_1' added successfully!" (yellow bulb icon) confirms initial addition, "Camera 'Camera_1' added successfully" (green checkmark) indicates validation, "Loaded 1 configured camera(s)" shows persistence across restarts, "Adding camera: Camera_1..." begins the cloud sync process, and "Camera Agent Ready" signals full operational status. These logs provide transparency into the agent's internal state machine, helping troubleshoot connectivity issues between local cameras, the go2rtc transcoding service, and the remote signaling server. The logging system tracks WebSocket connection state, TURN server reachability, camera stream health, and configuration changes—critical for diagnosing network issues in production deployments across 20+ client sites. The logs also record automatic reconnection attempts when network interruptions occur, ensuring the system self-heals without manual intervention.

4. Cloud Dashboard - Multi-Camera WebRTC Streaming

Description: The cloud dashboard interface displaying the 4-camera grid view with real-time WebRTC video streaming. The header shows "4-Camera Multi-Network WebRTC Streaming" with cloud server connection status. "Refresh Available Cameras" button queries the signaling server for updated camera lists across all connected sites. Camera Slot 1 (marked "LIVE" in green) displays an active stream from "Camera 1" showing a room interior—the video is being transmitted via WebRTC peer connection through the TURN relay server from a remote client site. The other slots (2, 3, 4 marked "READY" in red) await camera selection from dropdowns populated with all registered cameras across the network. Each slot includes Start/Stop controls, status indicators, and real-time connection logs showing WebRTC signaling state transitions: "Answer processed," "State: connecting," "ICE: connected," "Connected!" This dashboard operates entirely in the browser using JavaScript WebRTC APIs (RTCPeerConnection), establishing peer connections on-demand when users click "Start" to minimize bandwidth usage. The system automatically handles ICE candidate exchange through the signaling server, negotiates optimal connection paths (attempting direct peer-to-peer first, falling back to TURN relay if needed), and maintains stable <500ms latency streams. Multiple users can simultaneously view different camera combinations without impacting performance, as each viewer establishes independent WebRTC sessions.

Conclusion

This Cloud-Based WebRTC Camera Streaming System demonstrates a production-grade solution to one of the most challenging problems in distributed surveillance: secure, low-latency remote camera access without networking complexity. By eliminating port forwarding requirements and leveraging cloud-based TURN/STUN infrastructure, the system reduced deployment time per site from days to under 30 minutes while maintaining enterprise-level security through TLS encryption and isolated peer connections.

The architecture's manufacturer-agnostic approach via RTSP protocol support provides unparalleled flexibility—working seamlessly with Hikvision, Dahua, Axis, and other IP camera brands without vendor lock-in. The modular design with separate camera agents, signaling servers, and relay infrastructure enables independent scaling of each component based on load. Currently serving 20+ client sites with 50+ cameras, the system maintains 99.8% uptime and sub-500ms latency, validating the architectural decisions around WebRTC for real-time media transmission.

Future enhancements include implementing adaptive bitrate streaming based on network conditions, adding AI-powered motion detection and event triggering, developing mobile applications for iOS/Android using the same WebRTC infrastructure, and expanding the system to support two-way audio communication and PTZ (pan-tilt-zoom) control.
